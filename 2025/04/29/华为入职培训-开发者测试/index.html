1. 开发者测试概述

1.1 开发者测试基础概念

开发者测试(Developer Testing, DT) 是一种针对软件系统进行测试的方法，目的是确保软件系统的质量。其通常包括：

  • Unit Test：单元测试，对软件系统的最小单元进行测试，如函数、类、模块等。
  • Integration Test：集成测试，对软件系统的多个模块进行测试，以验证软件系统的功能。
  • 部分ST：系统测试，对软件系统的功能进行测试，以验证软件系统的质量。

1.2 可测试性

可测试性指系统或组件能够被有效测试的程度，包括：

  1. 可观察性：测试结果清晰可见；
  2. 可控性：能方便地设置测试条件；
  3. 可分解性：可隔离测试单个模块；
  4. 简单性：结构清晰，减少复杂依赖。
  5. 可隔离性：测试结果可独立验证；

高可测试性可提升缺陷发现效率，降低维护成本。

3. C/C++测试实现与执行

3.1 测试框架概述

3.2 gtest测试框架

GTest（Google Test）是 Google 开发的一个 C++ 测试框架，用于编写和运行单元测试。它是目前最流行的 C++ 测试框架之一，具有以下特点：

  1. 跨平台：支持 Windows、Linux、macOS 等多种操作系统
  2. 丰富的断言：提供多种断言宏来验证测试结果
  3. 测试发现：自动发现和运行测试用例
  4. 丰富的输出：提供详细的测试结果报告
  5. 死亡测试：支持测试程序是否按预期崩溃
  6. 参数化测试：支持使用不同参数运行同一测试
  7. 测试夹具：支持测试用例间的共享设置和清理

基本用法示例

常用断言宏

  • 基本断言：EXPECT_TRUE(condition), ASSERT_TRUE(condition)
  • 相等性检查：EXPECT_EQ(expected, actual), ASSERT_EQ(expected, actual)
  • 字符串比较：EXPECT_STREQ(str1, str2)

3.3 gMock基础知识

gMock 是 Google 提供的一个 C++ 测试框架，用于编写和使用 Mock 对象（模拟对象），通常与 Google Test（gtest） 结合使用，方便进行单元测试。它属于 Google C++ Testing Framework 的一部分。

gMock 的核心概念

  1. Mock 对象

    • 在单元测试中，某些真实对象（如数据库、网络服务）可能难以直接使用（速度慢、依赖复杂）。
    • Mock 对象模拟这些真实对象的行为，让测试更可控、更快速。

  2. 期望（Expectations）

    • 定义 Mock 对象的行为，例如：
      • 方法被调用时返回什么值？
      • 方法应该被调用多少次？
      • 调用时参数应该是什么？

  3. 断言（Assertions）

    • 验证 Mock 对象是否按预期被调用（例如：EXPECT_CALL）。
  • 模拟类和接口：通过 MOCK_METHOD 宏生成 Mock 类。
  • 设置期望：指定 Mock 方法的调用方式、参数、返回值等。
  • 参数匹配：支持灵活的参数匹配（如 _ 通配符、特定条件）。
  • 调用次数检查：验证方法是否被调用、调用次数是否符合预期。
  • 顺序控制（可选）：检查调用顺序是否满足要求。

gMock 的典型用途

  1. 隔离依赖：模拟数据库、网络、文件系统等外部依赖。
  2. 行为验证：确保某个方法被正确调用（如日志记录、回调触发）。
  3. 异常测试：模拟异常场景（如网络超时、磁盘满）。

安装与使用

  1. 安装：
    • 通过源码编译（Google Test/gMock 官方仓库）或包管理器（如 vcpkg、apt-get）。
  2. 集成：
    • 链接 gtest 和 gmock 库到测试项目。

优势

  • 易用性：通过宏快速定义 Mock 类。
  • 灵活性：支持复杂的参数匹配和调用控制。
  • 与 gtest 无缝集成：直接结合 TEST 和 ASSERT_* 使用。

自测题

  1. 语句覆盖（A）：

    • 要求每条语句至少执行一次。
    • 是最弱的覆盖准则，无法保证分支或条件的全面测试。

  2. 分支覆盖（B）：

    • 要求每个分支的真假两种情况至少执行一次。
    • 比语句覆盖强，但未考虑分支中多个条件的组合情况。

  3. 条件覆盖（C）：

    • 要求每个条件的真假两种情况至少执行一次。
    • 比分支覆盖更细致，但可能遗漏分支的整体真假组合。

  4. 路径覆盖（D）：

    • 要求覆盖程序中所有可能的路径（即语句、分支和条件的完整组合）。
    • 是最强的覆盖准则，能够全面测试代码的所有执行路径，但实现难度较高。
      正确答案是 D（路径覆盖），因为它提供了最全面的测试覆盖。
