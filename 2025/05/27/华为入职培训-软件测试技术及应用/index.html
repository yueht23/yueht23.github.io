2.3测试原则

  • 穷尽测试是不可能的
  • 杀虫剂悖论，反复使用相同的测试，可能会导致测试结果不准确
  • 测试显示缺陷的存在，但不能证明系统不存在缺陷
  • 测试本质上一种抽样检验
  • NO risk, No test
  • Boehm’s Law:在开发过程中越晚修正缺陷，代价越大
  • Absence of error谬误：认为“软件没有发现缺陷”等同于“软件没有缺陷”或“软件完全满足用户需求”。这种谬误忽略了测试的根本目标，可能导致对软件质量的错误信心。

2.4测试模型

  • T-IBO模型
  • Test Oracle（测试预言）Test Oracle 是软件测试中的一个关键概念，指一种能够判断被测系统的输出是否正确的机制或方法。它用于验证测试用例的预期结果是否与实际结果一致，从而确定测试是否通过,常见的有：
    • 确定性 Oracle（Deterministic Oracle）预期结果是明确且唯一的，如数学运算、协议响应等。
    • 启发式 Oracle（Heuristic Oracle）结果可能有一定灵活性，需通过规则或经验判断。示例：图像识别系统对某张图片的分类置信度 >90% 即算正确。
    • 统计 Oracle（Statistical Oracle）基于概率或统计规律验证结果，如性能测试中的吞吐量应在合理范围内。
    • 无 Oracle 问题（Oracle Problem）某些场景难以定义准确的 Oracle（如复杂 AI 系统、创造性生成内容），需依赖人工评估或替代方法。

2.5测试术语

  • EORROR/BUG/Failure
  • Verification/Validation

分类标准：

3.2黑盒测试设计技术

3.2.1 等价类（Equivalence Partitioning）

将输入数据划分为若干组，每组中的数据在测试中应产生相同输出。目的是通过减少测试用例数量覆盖更多场景。

  • 有效等价类：符合规范的输入
  • 无效等价类：非法或异常输入
    示例：测试输入1-100的文本框，可分为：
  • 有效：1-100之间的数字
  • 无效：小于1、大于100的数字、非数字字符,eg. 啤酒沙发饮料

3.2.2 边界值分析（Boundary Value Analysis）

专注于输入范围的边界条件，因为边界附近最容易出现错误。通常测试：

  • 最小值、略高于最小值
  • 正常值
  • 最大值、略低于最大值
    示例：同上1-100的输入框，测试：0,1,2,99,100,101

3.2.3 数据组合（Combinatorial Testing）

系统化组合多个输入参数，以检测参数间交互可能引发的缺陷。常用技术包括：

  • 全组合：覆盖所有可能（成本高）
  • 配对测试（Pairwise）：覆盖所有两两组合（高效）
    示例：测试登录功能（用户名类型×密码长度×验证码状态）
    有一些开源的工具如PICT 可以自动生成数据组合用例。

3.2.4 判定表（Decision Table）

以表格形式表示不同输入组合对应的系统行为，适合处理业务规则复杂的场景。包含：

  • 条件桩（输入条件）
  • 动作桩（预期输出）
  • 规则（条件与动作的对应关系）
    示例：机票折扣规则（会员等级×购票时间→折扣率）

3.2.5 判定点（Decision Point）

程序流程中可能产生不同分支的逻辑判断点。测试需覆盖所有可能路径。
示例：if-else、switch-case语句都是典型的判定点

3.2.6 PCT（Pairwise Combinatorial Testing）

一种优化的组合测试技术，通过数学算法（如正交阵列）保证所有参数的两两组合被覆盖，平衡测试效率与缺陷发现能力。
工具支持：如微软PICT、Hexawise等

3.2.7 状态机测试（State Transition Testing）

适用于具有明确状态转换的系统。通过：

  1. 绘制状态转换图
  2. 设计覆盖典型路径、异常路径的测试用例
    示例：ATM机状态流程（空闲→插卡→输入密码→选择服务→完成→退卡）

这些技术可单独使用或组合应用，根据系统特点（如输入维度、业务规则复杂度等）选择最合适的方法。实际测试中常配合因果图、场景法等技术增强覆盖度。

3.3白盒测试设计技术

白盒测试是一种软件测试方法，测试人员基于代码的内部结构、设计和实现来设计测试用例。与黑盒测试（仅关注输入输出）不同，白盒测试需要了解程序的内部逻辑，通常用于验证代码路径、条件分支、循环和数据结构等。

白盒测试的主要技术

  1. 语句覆盖（Statement Coverage）：确保代码中的每一行至少执行一次。
  2. 分支覆盖（Branch Coverage）：测试所有可能的条件分支（如 if-else）。
  3. 路径覆盖（Path Coverage）：覆盖代码中所有可能的执行路径。
  4. 条件覆盖（Condition Coverage）：测试布尔表达式的所有可能结果。
  5. 循环覆盖（Loop Coverage）：验证循环结构的正确性（如 for、while）。

常用的白盒测试工具

1. 静态代码分析工具（不运行代码）

  • SonarQube：检测代码质量、安全漏洞和代码异味。
  • Checkstyle（Java）：检查代码是否符合编码规范。
  • PMD（Java/JavaScript等）：识别潜在的错误和不良实践。
  • ESLint（JavaScript）：静态分析 JavaScript 代码。
  • Pylint（Python）：检查 Python 代码的错误和风格问题。

2. 动态代码分析工具（运行代码）

  • JUnit（Java）、NUnit（.NET）、PyTest（Python）：单元测试框架，用于白盒测试。
  • JaCoCo（Java）、Coverage.py（Python）：代码覆盖率工具，检查测试覆盖情况。
  • GDB（GNU Debugger）：调试工具，用于分析程序运行时的行为。
  • Valgrind（C/C++）：检测内存泄漏和性能问题。

3. 集成/高级白盒测试工具

  • Selenium（Web 自动化）：可与单元测试框架结合，进行白盒+黑盒混合测试。
  • Postman（API 测试）：用于白盒测试后端逻辑。
  • Klocwork（C/C++/Java）：静态分析 + 动态测试，用于企业级代码检查。

适用场景

  • 单元测试（开发者编写）
  • 代码审查
  • 安全漏洞检测
  • 性能优化分析
    白盒测试通常由开发人员或测试工程师执行，适合在开发早期发现逻辑错误，提高代码质量。
